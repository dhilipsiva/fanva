- **MCP spec facts (2025-06-18, from `modelcontextprotocol.io/specification/2025-06-18/basic/transports`):**
  - Streamable HTTP "replaces the HTTP+SSE transport from protocol version 2024-11-05"; the server "MUST provide a single HTTP endpoint path … that supports both POST and GET methods."
  - Per spec: "The client MUST include an `Accept` header, listing both `application/json` and `text/event-stream` as supported content types." (Servers enforcing this can return HTTP 406 / JSON-RPC -32600 if it is missing.)
  - After the `initialize` result, the client MUST send `notifications/initialized` (JSON-RPC notification, no id). Server MUST NOT send non-ping/logging requests before receiving it.
  - Per spec: "If the server accepts the input [a notification/response], the server MUST return HTTP status code 202 Accepted with no body. If the server cannot accept the input, it MUST return an HTTP error status code (e.g., 400 Bad Request)."
  - A JSON-RPC *request* POST → server returns EITHER `Content-Type: application/json` (one JSON object) OR `Content-Type: text/event-stream` (SSE stream). The client MUST support both.
  - Session: server MAY return `Mcp-Session-Id` on the InitializeResult response; if it does, the client MUST echo it on all subsequent requests. HTTP 404 on a request with a session id → session expired → restart with a new `initialize`. (Probing found no session header from jbotci — handle one if it appears.)
  - `MCP-Protocol-Version: 2025-06-18` header MUST be sent on all requests after initialization (HTTP transport).
  - **Tool-call result shape:** JSON-RPC `result` = `{ content: [ {type:"text", text} | image | audio | resource ], isError?: bool, structuredContent?: {} }`. Tool-execution failures are reported with `isError: true` INSIDE the result so the LLM can see and self-correct. Only "tool not found" / malformed / unsupported → a JSON-RPC protocol error object (e.g. `{code:-32602, message}`).
  - `tools/list` returns each tool as `{name, description, inputSchema}` where `inputSchema` is a JSON Schema — map it into each provider's tool/function-declaration format.
- **Provider tool-calling shapes (current, mid-2026):**
  - **Anthropic Messages API:** `tools:[{name, description, input_schema}]`; assistant returns `content` blocks `{type:"tool_use", id, name, input}` with `stop_reason:"tool_use"`; caller replies with a **user** message whose content has `{type:"tool_result", tool_use_id, content}` blocks (tool_result blocks MUST come first in the array). Headers: `x-api-key`, `anthropic-version: 2023-06-01`, and for browser calls `anthropic-dangerous-direct-browser-access: true` (without it the API returns `authentication_error: "CORS requests must set 'anthropic-dangerous-direct-browser-access' header"`).
  - **OpenAI / OpenRouter Chat Completions:** `tools:[{type:"function", function:{name, description, parameters}}]`; assistant `message.tool_calls:[{id, type:"function", function:{name, arguments}}]` where `arguments` is a **stringified JSON that must be `JSON.parse`d** (may be invalid JSON — validate); `finish_reason:"tool_calls"`; caller appends `{role:"tool", tool_call_id, content}`. OpenRouter base URL `https://openrouter.ai/api/v1`, identical Chat Completions schema (the top-level `function` wrapper is REQUIRED for Chat Completions, unlike the Responses API).
  - **Google Gemini:** `tools:[{functionDeclarations:[{name, description, parameters}]}]`; model returns a part `{functionCall:{name, args}}` (args is an object, not a string); caller sends a part `{functionResponse:{name, response}}`. Assistant role is `"model"` (not "assistant"); system prompt goes in top-level `systemInstruction`; the API is stateless — resend the whole history each turn.
